# Design: CycloneDX 1.7 support (issue #95)

**Date:** 2026-07-21
**Issue:** [#95 — Add CycloneDX 1.7 support via cyclonedx-go v0.11.0](https://github.com/OmniTrustILM/cbom-repository/issues/95)
**Status:** approved design, pre-implementation

## Goal

Let cbom-repository ingest, validate, store, and serve CycloneDX **1.7** CBOM documents alongside the existing **1.6** support, without regressing 1.6 behaviour.

## Scope decisions (fixed by product)

- Support **1.6 and 1.7 only.** Do not add 1.4/1.5.
- **No end-to-end verification against cbom-lens.** Issue #95's acceptance criterion "verified with CBOM Lens 1.7 output (depends on cbom-lens#175)" is moved to a **separate ticket**; this work carries no cross-repo dependency.

## Background: how versions flow today (verified in code)

- `internal/service/service.go:35-37` — `versionToEmbeddedFileMapping = {"1.6": "schemas/bom-1.6.schema.json"}` is the single version registry. `//go:embed schemas` (service.go:29-30) embeds the directory; `New()` (service.go:70-92) compiles every mapped schema at boot into `jsonSchemas map[string]*jss.Schema` (kaptinlin/jsonschema). A compile failure makes `New()` fail → the service does not start.
- HTTP negotiation: the schema version comes from the `Content-Type: application/vnd.cyclonedx+json; version=X` parameter; when omitted it defaults to `"1.6"` (`internal/http/mime.go:8`). `handlers.go:28-31` rejects unsupported versions (400) via `service.VersionSupported()` (derived from `jsonSchemas` keys). **Consequence: adding `"1.7"` to the mapping auto-enables `version=1.7` negotiation with no handler change.**
- Upload cross-check: the body is decoded into a `cdx.BOM` (cyclonedx-go); `uploadInputChecks`/`knownCdxVersion` (`upload.go:263-282`, hardcoded 1.0–1.6) assert `bom.SpecVersion` matches the declared version. Schema validation runs on the **original bytes** (`upload.go:73-83`).
- **cyclonedx-go is at v0.11.0 on `main`** (merged via PR #109; verified via the repo contents API). v0.11.0 adds the `SpecVersion1_7` enum and 1.7 struct fields (upstream PR #257). It does **not** expose the 1.7 JSON schema to consumers (only a URL string), so we must vendor our own schema file.
- Retrieval (`GetBOMByUrn`, byte-preserving), `CheckOnFetch` (generic JSON syntax check), search (S3 key/metadata based), and all of `internal/store` are **version-agnostic** — confirmed, no changes needed.

## Design

### 1. Vendor and embed the 1.7 schema
Add `internal/service/schemas/bom-1.7.schema.json` (official CycloneDX 1.7 JSON schema). The existing `//go:embed schemas` directive already covers it. The 1.6 schema carries two **unresolved external `$ref`s** — `spdx.schema.json` (bom-1.6:1253) and `jsf-0.82.schema.json` (bom-1.6:~5040); kaptinlin/jsonschema tolerates unresolved refs (SPDX-license-id and JSF-signature sub-validation are silently not enforced). The 1.7 schema has the same refs. We **mirror 1.6 exactly and leave them unresolved** — closing that gap is pre-existing and out of scope. (Empirically confirmed: the official 1.7 schema compiles cleanly with the repo's kaptinlin validator.)

### 2. Register 1.7
Add `"1.7": "schemas/bom-1.7.schema.json"` to `versionToEmbeddedFileMapping` (service.go). This enables boot-compile, `VersionSupported("1.7")`, and `version=1.7` HTTP negotiation.

### 3. Version cross-check
Add a `"1.7" → cdx.SpecVersion1_7` case to `knownCdxVersion` (upload.go). Additionally, make the "schema validator missing" branch (`upload.go:73-78`) return `ErrValidation` (→400) rather than a bare error (→500), so a direct service call with a version that passes `knownCdxVersion` but has no compiled schema cannot produce a 500.

### 4. Configurable default BOM version
Replace the `defaultBOMVersion = "1.6"` constant (`internal/http/mime.go:8`) with a config value wired through the existing envconfig `Config` (like `APP_CHECK_ON_FETCH`): `APP_DEFAULT_BOM_VERSION`, default `"1.6"`. `CheckContentType` becomes a method/closure over the configured default (or the default is injected), so an omitted `version=` media-type parameter uses the configured value. **Fail-fast guard:** at startup, assert the configured default is a supported version (present in `versionToEmbeddedFileMapping`); otherwise `New()` (or server construction) returns an error so a misconfigured deployment does not boot. Keeps default `1.6` unless an operator opts into `1.7`.

### 5. Fix decode-error misclassification (bug found in recon)
A body whose `specVersion` cannot be decoded returns the cyclonedx-go error **unwrapped** (`upload.go:64-67`); the handler classifies it as **500** because only `ErrValidation`/`ErrAlreadyExists`/`*http.MaxBytesError` are special-cased. With v0.11.0 this now only affects versions the library doesn't know at all (e.g. `"1.8"`, garbage) — the 1.6/1.7 declared-version mismatch cases already return a clean 400 via the cross-check (see §7). Wrap decode failures as `ErrValidation` so an unknown/garbage `specVersion` returns **400** — **but preserve `*http.MaxBytesError`**: guard with `errors.As(&maxErr)` and return it unwrapped (or otherwise keep it reachable via `errors.As`), so oversized bodies still return **413**, not 400. `handlers.go:35-39` checks `MaxBytesError` before `ErrValidation`; `handlers_test.go:756` guards the 413 path.

### 6. Crypto-stats: confirmed no-op (deep-verified)
Verified at both the Go-struct and JSON-schema level (cyclonedx-go v0.10.0 vs v0.11.0, and `bom-1.6.schema.json` vs `bom-1.7.schema.json`): the `CryptoAssetType` enum is byte-identical (the same 4 values), the `cryptographic-asset` component type is unchanged, and `cryptoProperties.assetType` is identical in both schemas. `stats.go:41-74` discriminates only on `component.Type == cryptographic-asset` and the 4 `AssetType` buckets, incrementing `Total` before the switch — all of which are unchanged in 1.7. 1.7's crypto-registry additions live entirely **inside** the sub-structs (new protocol subtypes, `algorithmFamily`, certificate state/extensions, related-crypto-asset refs) which `stats.go` never inspects. **Conclusion: no code change; no new counter** (which would also change the fixed OpenAPI response shape, `api/openapi.yaml:369-388`). A 1.7 crypto-stats smoke test asserts parity. Exposing the richer 1.7 crypto dimensions in stats is net-new feature work → separate ticket (see out of scope).

### 7. Version cross-check behaviour (document + test the real semantics)
The JSON schema does **not** constrain `specVersion` (no `const`/`enum`); the declared version is enforced solely by `knownCdxVersion`/`uploadInputChecks` (`upload.go:244-282`), which runs before schema validation. Verified outcomes to lock in with tests and docs:
- **Content-Type `version=1.7` + body `specVersion:"1.7"` with 1.6-shaped content → accepted** (correct: 1.7 is a backward-compatible superset; a real 1.6 crypto BOM validates against both schemas).
- **`version=1.6` + body `specVersion:"1.7"` → 400** `ErrValidation` ("required version 1.6"), caught at the cross-check before schema validation.
- **`version=1.7` + body `specVersion:"1.6"` → 400** `ErrValidation` (symmetric).
This behaviour is already correct once §2/§3 land; the work here is explicit regression tests + a docs note that the repo validates the *declared* version, not the document's structural shape.

### 8. Docs
- `README.md` / `api/openapi.yaml`: state support for 1.6 **and** 1.7; document that a 1.7 upload requires `Content-Type: application/vnd.cyclonedx+json; version=1.7` (an omitted version defaults to 1.6 and a 1.7 body would be rejected as a version mismatch — intended behaviour, no body-sniffing).
- Reconcile the OpenAPI mismatch: unsupported version returns **400** in code/tests (`handlers.go:28-31`, `handlers_test.go:249-254`) but OpenAPI documents **415** (`api/openapi.yaml:55-56`).
- Fix the stale README claim (`README.md:46`) that the service auto-detects the version — it does not.

## What explicitly does NOT change (YAGNI)

Retrieval (`GetBOMByUrn`), `CheckOnFetch`, search, the entire `internal/store` layer; no body-version sniffing; no new crypto counters; no embedding of spdx/jsf sub-schemas.

## Testing (TDD)

1. **Fidelity test (highest value):** the serial-number-missing upload paths (`upload.go:147,188`) *re-encode* the decoded `cdx.BOM` and store that as the served document, while schema validation runs on the original bytes. A 1.7 document carrying 1.7-specific fields must survive encode→decode→encode without field loss. Test this on the SN-missing path (guards against silent data loss if the struct model is incomplete).
2. Upload happy-path for a valid 1.7 doc with `version=1.7`; schema-rejection for an invalid 1.7 doc.
3. Version cross-check (§7): `version=1.6` + body `specVersion:1.7` → 400; `version=1.7` + body `specVersion:1.6` → 400; and a 1.6-shaped body honestly declaring `specVersion:1.7` under `version=1.7` → accepted.
4. Regression: oversized body still → 413 (guards the decode-error-wrapping change).
5. Regression: unknown/garbage `specVersion` (unknown to v0.11.0) → 400 not 500.
5a. Configurable default version: with `APP_DEFAULT_BOM_VERSION=1.7`, an upload omitting the `version=` param is treated as 1.7; an unsupported configured default fails startup fast.
6. `TestNewFunc`: `SupportedVersion()` → `["1.6","1.7"]`; `VersionSupported("1.7")` true.
7. Extend `TestKnownCdxVersion*` with the 1.7 case.
8. 1.7 crypto-stats smoke test (parity with 1.6 counting).
9. Parameterize `minimalBOMJSON` (currently hardcodes `specVersion:"1.6"`, single-digit version via `string(rune('0'+version))`) to accept a spec version.
10. Boot-compile smoke test (covered by `TestNewFunc` constructing the real `Service` via `New()`).

## Out of scope → separate tickets

1. End-to-end verification with CBOM Lens 1.7 output (cbom-lens#175). *(explicitly split out)*
2. Embedding `spdx.schema.json` / `jsf-0.82.schema.json` to enforce license-id and signature sub-schemas (pre-existing gap, affects 1.6 too).
3. Exposing 1.7's richer crypto-registry dimensions in crypto-stats (protocol subtypes, algorithm family, certificate state, related-crypto-asset counts). Net-new feature; the current 4-bucket counting is unchanged and correct for 1.7.

## Implementation order

0. Sync local to `origin/main`; assert `go.mod` shows cyclonedx-go **v0.11.0** (already merged via #109 — verify, don't `go get`). *(done)*
1. Vendor + embed `bom-1.7.schema.json`.
2. Register `"1.7"` in the mapping.
3. `knownCdxVersion` 1.7 case + missing-validator→`ErrValidation`.
4. Decode-error wrapping fix (MaxBytesError-safe).
5. Configurable default version (`APP_DEFAULT_BOM_VERSION`) + fail-fast guard.
6. Tests (fidelity first, then cross-check, regressions, config, parity).
7. Docs (README + OpenAPI: versions, `version=` param requirement, 400-not-415 reconciliation, drop stale auto-detect claim).

The dependency (step 0) gates everything; `cdx.SpecVersion1_7` won't compile without it.
