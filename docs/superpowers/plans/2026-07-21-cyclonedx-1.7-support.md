# CycloneDX 1.7 Support Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let cbom-repository accept, validate, store, and serve CycloneDX **1.7** CBOM documents alongside 1.6, without regressing 1.6 behaviour.

**Architecture:** Additive wiring in the `service` and `http` layers. Validation stays "schema-per-declared-version": embed the official 1.7 JSON schema, register it, and teach the version cross-check about 1.7. Retrieval/search/store are version-agnostic and untouched. Two correctness fixes ride along: decode errors become `400` (not `500`) while preserving the `413` oversized path, and the default upload version becomes env-configurable.

**Tech Stack:** Go 1.26, `github.com/CycloneDX/cyclonedx-go v0.11.0` (struct decode/encode + `SpecVersion1_7`), `github.com/kaptinlin/jsonschema` (JSON-schema validation), `envconfig`, `gorilla/mux`, `testify` + `go.uber.org/mock` (gomock).

## Global Constraints

- Support versions **1.6 and 1.7 only**. Do not add 1.4/1.5 support (the existing `knownCdxVersion` 1.0–1.5 cases stay for cross-check purposes but no schema is registered for them).
- `github.com/CycloneDX/cyclonedx-go` MUST be **v0.11.0** (already on `main` via #109). `cdx.SpecVersion1_7` does not exist below it.
- The 1.7 JSON schema is sourced from the pinned module: `$(go env GOMODCACHE)/github.com/!cyclone!d!x/cyclonedx-go@v0.11.0/schema/bom-1.7.schema.json`.
- TDD: write the failing test first, watch it fail, implement minimally, watch it pass, commit. One logical change per commit.
- Branch: `feature/cyclonedx-1.7-support` (already created). The `main` ruleset requires squash-merge, 1 code-owner approval, all of `license/cla`/`Check`/`Check for Copilot Review`/`SonarCloud Code Analysis` green, resolved review threads, and passing CodeQL — so keep commits clean and the diff focused.

### Task 3: Classify undecodable specVersion as 400, preserving the 413 oversized path

**Files:**
- Modify: `internal/service/upload.go:64-67` (decode-error wrapping)
- Test: `internal/service/upload_test.go` (new unknown-version test); `internal/http/handlers_test.go` (existing oversized/413 test must still pass)

**Interfaces:**
- Consumes: `ErrValidation` (service.go:24).
- Produces: a body with a `specVersion` unknown to cyclonedx-go (e.g. `"1.8"`, garbage) makes `UploadBOM` return an error satisfying `errors.Is(err, ErrValidation)`; an oversized body still surfaces `*http.MaxBytesError` via `errors.As`.

Rationale: `fmt.Errorf("%w: %w", ErrValidation, err)` wraps **both** sentinels (Go ≥1.20 multi-`%w`; this repo is Go 1.26). The HTTP handler (`handlers.go:37-54`) checks `errors.As(&maxErr)` *before* `errors.Is(ErrValidation)`, so oversized bodies stay `413` while other decode failures become `400`.

The upload path re-encodes the decoded struct on the serial-number-missing and version-missing branches, while schema validation runs on the *original* bytes — so a dropped field would pass validation yet be lost in the stored/served document. This test pins that exact risk at the decode/encode boundary.
